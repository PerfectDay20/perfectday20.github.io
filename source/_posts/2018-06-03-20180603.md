1. 与Spark不同，Scala中集合的transform操作都是strict的，所以都会产生中间集合，除了 `view` 和 `stream`。在两种情况下可以考虑使用 `view`：
  a. 性能；大集合才明显，小集合反而会因为构造闭包而变慢；或者可以提前停止的，比如`take` ，`first` 之类的，比如
  ```scala
  (1 to 1000000000).filter(_ % 2 == 0).take(10).toList
  (1 to 1000000000).view.filter(_ % 2 == 0).take(10).toList
  (1 to 1000000000).view.filter(_ % 2 == 0).sum
  ```
  b. 作用于可变集合；对集合的一部分生成 `view` 的 `slice`，然后套用函数修改，这样有助于模块化，比如下面例子，选取范围和取反操作互相分离
  ```scala
  val arr = (0 to 9).toArray
  val subArr = arr.view.slice(3, 6)
  def negate(xs:mutable.Seq[Int]) = for(i<-xs.indices) xs(i) = -xs(i)
  negate(subArr)
  ```
  使用时注意，由于lazy evaluation的作用，要不使用纯函数式没有副作用的集合转换，要不使用明确的可变集合修改操作，不要既创建集合又产生副作用，比如：
  ```scala
    (1 to 10).view.map(i => Future(println(i))) // 并不会打印东西
  ```
2. Spark处理数据时，对于一些因为格式不正确等需要剔除的数据时，可以使用Accumulator进行统计
3. 每个stage中的core与task比例一般可设为1:4； [这篇文章](https://blog.scottlogic.com/2018/03/22/apache-spark-performance.html)中遇到的问题是对`Boolean`值进行了 `repartition`，导致只有两个分区有数据，最后改成了 `write` 之后 `partitionBy` ，解决了并发问题。问题很常见，作者也没有考虑最后保存文件数，总体收获一般
4. Bash `set -- a b c` 可以重设 `$1`~`$3` 的值
5. Bash `[[ ]]` 中使用 `==` 和 `=~` 时，右边参数被引号包围的部分，就是按照字面进行对比，即
```bash
[[ "foo" == "foo" ]] -> true
[[ "foo" == foo ]] -> true
[[ "foo" == f* ]] -> true
[[ "foo" == "f*" ]] -> false
```

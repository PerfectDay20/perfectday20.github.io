Both Parquet and ORC are very useful columnar storage formats, and they can be easily used in big data systems, like hive or Spark, without knowing their internal implementation. But when you want to use them in POJO, things are different.

For [Parquet](https://github.com/apache/parquet-mr), there are many official support for other formats, like protobuf. You can write a protobuf file, generate Java class, then use it to create a Parquet file.

For [ORC](https://github.com/apache/orc), there is no such thing, only in its [documentation](https://orc.apache.org/docs/core-java.html), there is a tutorial to help you write your custom `Writer`. Yes, this maybe more efficient for CPU, but the cost is that you have to know much more details about ORC implementation and how to control its API, decide whether this column contains `Null` or not, whether the value is repeating or not... It's very error prone, and it's just the beginning of the nightmare, until you want to create a writer for another data object, and start containing `Map` in ORC...

After searching other methods to write ORC in POJO, I found the most suitable way for me is to use the [mapreduce API](https://orc.apache.org/docs/mapreduce.html).

```java
import java.io.IOException;
import org.apache.hadoop.conf.Configuration;
import org.apache.hadoop.fs.Path;
import org.apache.hadoop.io.IntWritable;
import org.apache.hadoop.io.LongWritable;
import org.apache.hadoop.io.Text;
import org.apache.orc.CompressionKind;
import org.apache.orc.OrcFile;
import org.apache.orc.OrcFile.WriterOptions;
import org.apache.orc.TypeDescription;
import org.apache.orc.Writer;
import org.apache.orc.mapred.OrcMap;
import org.apache.orc.mapred.OrcStruct;
import org.apache.orc.mapreduce.OrcMapreduceRecordWriter;

public class MapReduce {

    public static void main(String[] args) throws IOException {

        Path path = new Path("test.orc");
        Configuration conf = new Configuration();
        TypeDescription schema = TypeDescription
                .fromString("struct<name:string,age:int,cu:map<string,bigint>,cr:map<string,map<string,bigint>>>");
        WriterOptions options = OrcFile.writerOptions(conf)
                .setSchema(schema)
                .overwrite(true)
                .compress(CompressionKind.SNAPPY);

        Writer writer = OrcFile.createWriter(path, options);
        OrcMapreduceRecordWriter<OrcStruct> recordWriter = new OrcMapreduceRecordWriter<>(writer);

        OrcStruct struct = (OrcStruct) OrcStruct.createValue(schema);

        OrcMap<Text, LongWritable> cu = new OrcMap<>(
                TypeDescription.createMap(TypeDescription.createString(), TypeDescription.createLong()));
        cu.put(new Text("hello"), new LongWritable(10));
        cu.put(new Text("world"), new LongWritable(10));

        OrcMap<Text, OrcMap<Text, LongWritable>> cr = new OrcMap<>(TypeDescription.createMap(
                TypeDescription.createString(),
                TypeDescription.createMap(TypeDescription.createString(), TypeDescription.createLong()))
        );

        OrcMap<Text, LongWritable> crInner1 = new OrcMap<>(
                TypeDescription.createMap(TypeDescription.createString(), TypeDescription.createLong()));
        OrcMap<Text, LongWritable> crInner2 = new OrcMap<>(
                TypeDescription.createMap(TypeDescription.createString(), TypeDescription.createLong()));
        crInner1.put(new Text("1"), new LongWritable(1));
        crInner2.put(new Text("2"), new LongWritable(2));
        cr.put(new Text("crInner1"), crInner1);
        cr.put(new Text("crInner2"), crInner2);

        struct.setAllFields(new Text("world"), new IntWritable(12), cu, cr);

        recordWriter.write(null, struct);
        recordWriter.close(null);
    }
}
```



When using this method, a CRC file will be written alongside the ORC file. To disable this behavior, we can extend the `Path` class:

```java
import java.io.IOException;
import org.apache.hadoop.conf.Configuration;
import org.apache.hadoop.fs.FileSystem;
import org.apache.hadoop.fs.Path;

public class NoCheckPath extends Path {

    public NoCheckPath(String pathString) throws IllegalArgumentException {
        super(pathString);
    }

    @Override
    public FileSystem getFileSystem(Configuration conf) throws IOException {
        FileSystem fs = super.getFileSystem(conf);
        fs.setWriteChecksum(false);
        return fs;
    }
}
```



A few disadvantages of this method:

- Import mapreduce package, may be misleading because we don't use MapReduce
- We have to convert every data type to its corresponding `Writable`, like `String` to `Text`, `int` to `IntWritable`
-  We have to write different writers for each data object type

But it's way easier than using the `*ColumnVector` API, and that's enough for me.

----

Closing note:

A library's API will influence programmer's decision greatly. Make it easy to use, without needing to know much about the internal.

20180618 阿里Java开发手册

从阿里Java开发手册学到的：

  1. 【强制】抽象类命名使用 Abstract 或 Base 开头（经常缩写成Abs或不写）
  2. 【强制】POJO 类中布尔类型的变量，都不要加 is 前缀，否则部分框架解析会引起序列化错误（比如jackson中没有域的getter会反序列化错误）
  3. 【强制】杜绝完全不规范的缩写，避免望文不知义（比如Abs）
  4. 【参考】枚举类名建议带上 Enum 后缀（有时不带）
  5. 【推荐】 类内方法定义的顺序依次是:公有方法或保护方法 > 私有方法 > getter/setter 方法（共有方法和依赖的私有方法在一起）
  6. 【推荐】集合初始化时，指定集合初始值大小（从不指定）
  7. 【推荐】除常用方法(如 getXxx/isXxx)等外，不要在条件判断中执行其它复杂的语句，将复杂逻辑判断的结果赋值给一个有意义的布尔变量名，以提高可读性（经常直接写）

20210329 update:

再回头看这些条目, 或者规定, 有点过于“迂腐”, 感觉对于刚学习编程的或者对于英语不好的开发者可能还有点用处.

代码是给人看的, 最重要的是可读性, 如果心里没有底, 则可以找一个规范跟着来, 当形成了自己的理解后, 则对各种规则就有了自己的判断了. 比如命名方面, EventType 和 EventTypeEnum 的可读性对比显而易见. 类内方法的顺序, 也应该按照读代码时候, 根据调用顺序, 或者其他逻辑规则放在一起, 如果强行按照各种方法可见性的顺序, 则读的时候跳来跳去, 影响效率.

20180617 按小时分区

Spark中将数据按小时分区，且希望每个文件大小平均，基本思路是先统计出各个小时的条目数，然后根据输入的总文件个数，算出每小时应该有几个分区，然后将数据repartition到各个分区。而repartition有两种方法：

1. 在Dataset范围内操作

给Dataset增加一个分区列，根据原来的hour列生成hour_partition列，比如一天数据需要100个分区，而20点这一小时计算出需要5个分区，则hour=20的数据的新增hour_partition列的值为20_0,20_1 … 20_4 平均分，然后利用Dataset.repartition(100, ‘hour_partition)即可。

但是这样操作会导致最终分区数少于设定值，因为Dataset.repartition是使用hash分区，虽然hour_partition列的这些值的hash一般不同，但是由于最后是对分区数求余，所以很容易出现一个分区包含多个小时的数据（这样不会影响最终文件数，但影响写入时间）和一个分区包含一个小时的不同hour_partition值的情况（这样会减少最终文件数，使得一个文件是其它文件的多倍大小）

2. 转换为RDD操作

由于Dataset的repartition无法使用自定义的Partitioner，所以可以先转换为RDD[hour_partition, Row]，然后使用 rdd.partitionBy(partitioner)，再转换回Dataset即可。这样最终文件数是与设定严格一致的。

虽然进行了Dataset与RDD的互相转换，但与方法1的计算时间在误差范围内

  1. Scala for 语句是通过编译成 map flatMap withFilter foreach 来实现的，所以拥有这些方法的类都可以用 for；withFilter 是 non-strict 的，也就是不生成中间集合；Java中的 for 是编译成 Iterator 或数组下标；Python中是 __getitem__ __iter__ __next__

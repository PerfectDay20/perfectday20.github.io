20180526 Early Definition

当子类覆盖父类的val且该val用于初始化父类时，会因为初始化顺序的不同产生问题。
比如上面这个例子，目的是为了让Ant的env长度为2，但是初始化的顺序是：

  1. 父类range设为10，生成getter和setter
  2. 初始化env，调用range的getter，因为被子类覆盖，调用子类的range getter
  3. 由于子类range未初始化，值为0，getter返回0，则父类的env长度初始化为0
  4. 初始化子类range为2

这样在生成Ant实例时，env长度就变为了0

解决方案：

  1. final val，不允许覆盖
  2. lazy val，但效率低
  3. early definition syntax：class Ant extends{override val range = 2} with Creature（由于trait没有构造参数，所以该语法经常用于覆盖父trait中的域，如class A extends {override val f = 1} with B）

20180523 Spark 写文件冲突

收获

  1. 两个spark任务向同一个目录中写文件，会造成冲突，导致任务失败；而采用之前多线程的方式，向一个父路径中利用partitionBy写文件，会导致文件结果缺失。这两个问题都是由于多个任务同时使用在父目录下的_temporary，造成冲突。

  3. 不能overwrite写到source路径中，因为开始执行的时候，source就被清空导致无法读取数据；和hive不同

20180522 Spark与多线程

收获

  1. 在spark driver中也可以使用thread pool来并行处理

    1. 因为spark是并行处理数据的，所以在使用spark的时候就很容易忘掉java本身的多线程；两者结合，在一些数据稍微倾斜的stage的最后一个task，和在保存文件需要长时间运行时，其它并行的task可以获得计算资源。在处理大量重复数据时很有效，比如连续处理100天的数据，则可以开一个fixedThreadPool来处理，比手动分隔日期启动多个spark任务要方便

  3. join->repartition(200)->sortWithPartitions->coalesce(5).write() 导致join发生在默认设置的spark.sql.shuffle.partitions的997个节点，奇怪

20180521 shuffle

收获

  1. Generators are not supported when it’s nested in expressions, explode 外不能嵌套函数，比如在SQL中使用split(explode(row), '_')
  2. shuffle操作耗时，除了因为网络传输外，还因为要写入磁盘
  3. 要减少shuffle耗时
    a. 减少shuffle的数据量，在本地先进行处理
    b. 减少创建不必要的对象，比如输入输出类型不一样时，不要将所有输入对象都转成输出对象，然后reduceByKey，而要使用aggregateByKey等操作；比如<A, B> -> <A, Set<B>> -> reduceByKey，创建了大量set对象
    c. 警惕逆向操作，比如先flatMap，join，然后又groupBy；可以用cogroup代替
    d. 相同partitioner，相同分区数，可以减少一次shuffle
  4. 生成stage边界的方法，一般都带有numPartitions参数，可以调节分区数；比如JavaRDD的groupBy，repartition，sortBy，substract；JavaPairRDD的aggregateByKey，cogroup，combineByKey，distinct，groupByKey，join，sortByKey，reduceByKey等

20180520 Parquet

收获

  1. Dataset orderBy = sort, 都是全局排序
  2. Dataset.write().sortBy 是分区内排序，但是现在还不支持写入Parquet
  3. 所以要减少体积的话，使用Dataset.orderBy(col1...).coalesce(n).write().parquet()，或者Dataset.coalesce(n).sortWithinPartition(col1...).write().parquet()，但是后者使用了coalesce后，并行任务数减少为n，所以运行很慢，所以建议使用前者
  4. Parquet官方建议row group size和hdfs block size相等，这样可以保证高读取速度，并且重建数据不需要读取其它block，减少了网络传递。但是大的row group size会导致高内存消耗。所以一个hdfs block里包含整数个row group也是可取的。http://ingest.tips/2015/01/31/parquet-row-group-size/
  5. 基于初步试验，根据cardinality大的列排序存储后的paruet文件更小

20180519

收获

  1. 当存储格式为Parquet，某一列全是null且无法推测类型时，比如使用selectExpr(" null as row1 ")，会导致错误Unsupported data type NullType，因为Parquet没有NULL类型，这时候可以使用"cast(null AS string) row1"设置该列的类型
  2. 尽量用内置方法而不是UDF，因为UDF对Catalyst不够透明，也就不能执行更有效的优化；UDF失败则任务失败，而内置方法则返回NULL;使用Python写Spark SQL时会经过Catalyst统一分析生成执行代码，但是假如使用了Python写的UDF，则会强迫JVM与Python序列化通信，增加运行时间；解决方法是……用Java或Scala写UDF。https://blog.cloudera.com/blog/2017/02/working-with-udfs-in-apache-spark/
  3. Parquet不同压缩格式对比，同一份数据分成100个分区，其中sorted两项是根据数据中两列sorted处理。根据结果发现，即使用了压缩体积最小的gzip，使用sorted后也能使体积缩小很多。
  4. 设置Spark的Parquet dict filter暂时还没发现什么效果
